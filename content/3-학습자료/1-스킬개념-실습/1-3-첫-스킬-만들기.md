# 1.3 첫 스킬 만들기

> skill-creator 스킬을 사용해서 나만의 첫 스킬을 만들어봅니다.

---

## 🎯 이번 시간 목표

- [ ] skill-creator로 스킬 생성하기
- [ ] 생성된 SKILL.md 확인하기
- [ ] 스킬 테스트해보기

---

## skill-creator란?

**스킬을 만드는 스킬**입니다.

| 직접 만들기 | skill-creator 사용 |
|------------|-------------------|
| SKILL.md 문법 공부 필요 | 대화하면서 자동 생성 |
| frontmatter 형식 외워야 함 | 질문에 답하면 완성 |
| 오타나면 에러 발생 | 검증된 구조로 생성 |

> 💡 **비유**: 웹사이트 만들 때 HTML 직접 쓰기 vs 노션/위즈/프리즘 사용하기

---

## 💡 스킬 만드는 두 가지 방법

| 방법 | 설명 |
|------|------|
| **방법 1** | skill-creator에게 "이런 스킬 만들어줘" → 티키타카하며 완성 |
| **방법 2** ⭐ | 작업하다가 → "이 과정을 스킬로 만들어줘" |

### 강사 추천: 방법 2

처음부터 "스킬 만들어야지!" 하기보다...

```
1. 평소처럼 Claude에게 작업 시킴
   ↓
2. "어? 이거 반복되겠는데?"
   "어? 이거 다른 팀원도 쓰면 좋겠는데?"
   ↓
3. "지금까지 한 이 작업을 스킬로 만들어줘"
   ↓
4. Claude가 SKILL.md 자동 생성!
```

> 💡 **실제로** 강사가 만든 스킬 대부분이 이 방식으로 탄생했습니다.

### 오늘 실습은 방법 1로!

일단 **skill-creator와 대화하면서 만드는 방법**을 먼저 익혀볼게요.
나중에 실제 업무하다가 "이거 스킬로 만들어줘"라고 해보세요!

---

## 실습: 첫 스킬 만들기

### Step 0: 실습 폴더 준비

스킬을 테스트하려면 **작업할 폴더**가 필요합니다. 간단한 폴더를 하나 만들어서 VSCode에서 열어볼게요.

**폴더 만들기**

1. **Finder** 열기
2. **Documents**(문서) 폴더로 이동
3. 우클릭 → **새로운 폴더** 만들기
4. 이름: `skill-test` 입력

```
Documents/
└── skill-test/    ← 여기서 실습합니다
```

**VSCode에서 열기**

1. VSCode **왼쪽 상단** 메뉴에서 `파일` 클릭
2. `폴더 열기...` 클릭
3. 방금 만든 `skill-test` 폴더 선택 → `열기`

```
VSCode 왼쪽 상단 메뉴:

파일(F)
├── 새 파일
├── 폴더 열기...   ← 이걸 클릭!
├── ...
```

**Claude 익스텐션 열기**

| Mac | Windows |
|-----|---------|
| `Cmd + Shift + Esc` | `Ctrl + Shift + Esc` |

오른쪽에 채팅창이 나타나면 준비 완료!

---

### Step 1: skill-creator 실행

채팅창에 입력:

```
/skill-creator
```

Claude가 몇 가지 질문을 합니다. 아래처럼 답해보세요.

---

### Step 2: 질문에 답하기

**질문 1: "어떤 스킬을 만들고 싶으세요?"**

예시 답변:
```
회의록 요약 스킬을 만들고 싶어요.
긴 회의록을 주면 핵심 결정사항과 액션아이템만 뽑아주는 거예요.
전역 스킬로 만들어줘.
```

> 💡 **"전역 스킬로 만들어줘"** 를 꼭 붙이세요!
> 전역 = 어떤 프로젝트에서든 사용 가능. 안 붙이면 현재 프로젝트에서만 사용됩니다.

**질문 2: "어떤 상황에서 이 스킬을 사용하나요?"**

예시 답변:
```
- 팀 회의 후 회의록 정리할 때
- 길게 적힌 미팅 노트 받았을 때
- "회의 요약해줘", "액션아이템 뽑아줘" 라고 말할 때
```

**질문 3: "결과물은 어떤 형태가 좋을까요?"**

예시 답변:
```
마크다운으로:
- 한 줄 요약
- 핵심 결정사항 (불릿)
- 액션아이템 (담당자, 마감일 포함)
```

---

### Step 3: 생성된 스킬 확인

Claude가 SKILL.md 파일을 만들어줍니다.

**확인할 것:**

1. **파일 위치**: `~/.claude/skills/meeting-summary/SKILL.md` (전역 스킬 폴더)
2. **frontmatter 확인**:
   ```yaml
   ---
   name: 회의록 요약
   description: 긴 회의록을 핵심 결정사항과 액션아이템으로 요약합니다.
     "회의 요약해줘", "액션아이템 뽑아줘" 라고 말하면 자동 실행됩니다.
   ---
   ```
3. **본문**: 스킬이 따라야 할 절차가 적혀있음

---

### Step 4: 스킬 테스트

**테스트 1**: 스킬 목록에서 확인

Claude에게:
```
사용 가능한 스킬 보여줘
```
→ 방금 만든 스킬이 목록에 보이면 성공!

> 💡 터미널에서는 `/skills` 명령어로도 확인할 수 있어요.

![](../../../public/images/CleanShot%202026-02-22%20at%2004.18.38@2x.png)

**테스트 2**: 테스트용 회의 스크립트 준비

실제 회의 녹음을 텍스트로 변환한 것처럼 만들어둔 예시입니다.
아래 내용을 **복사**하세요:

```markdown
# 프로덕트팀 주간 회의 - 2026.02.20 (목) 14:00

참석자: 김서연(PM), 이준호(디자인), 박민수(개발), 최하늘(마케팅)

---

김서연: 자 그럼 시작할게요. 오늘 안건 세 가지인데, 첫 번째가 신규 대시보드 일정 확인이고, 두 번째가 고객 피드백 대응, 세 번째가 마케팅 랜딩페이지 건이에요.

김서연: 먼저 대시보드 건. 이준호 님, 디자인 진행 상황 어때요?

이준호: 네, 와이어프레임은 다 끝났고요, 지금 UI 컴포넌트 작업 중이에요. 근데 차트 라이브러리를 뭘 쓸지 개발팀이랑 확인이 필요해서... 민수 씨, recharts랑 chart.js 중에 어떤 게 나을까요?

박민수: 저희 기존에 recharts 쓰고 있어서 recharts가 좋을 것 같아요. 새로 도입하면 번들 사이즈도 커지고.

이준호: 알겠습니다. 그러면 recharts 기준으로 작업할게요. 디자인은 다음주 수요일까지 나올 수 있어요.

김서연: 좋아요. 그러면 개발 착수는 디자인 나온 후에? 민수 씨 일정은요?

박민수: 디자인 받으면 한 2주 정도 잡아야 할 것 같아요. 그런데 지금 결제 모듈 버그가 있어서 그것부터 처리해야 돼요. 이번 주 내로 핫픽스 나갈 예정이에요.

김서연: 결제 버그가 우선이죠. 핫픽스 끝나고 대시보드 들어가면 되겠네요. 그러면 3월 첫째 주 출시 목표로 잡을까요?

박민수: 네, 가능할 것 같습니다.

김서연: 두 번째, 고객 피드백 건. 지난주에 엔터프라이즈 고객 세 곳에서 CSV 내보내기 기능 요청이 왔어요. 현재는 PDF만 되잖아요.

박민수: CSV 내보내기는 기술적으로 어렵진 않은데, API 쪽 작업이 한 3일 정도 필요해요.

김서연: 대시보드랑 겹치지 않게 일정 조율해야겠네요. 대시보드 끝나고 바로 들어가도 될까요?

박민수: 네, 그렇게 하죠. 3월 둘째 주쯤이면 될 것 같아요.

김서연: 마지막, 마케팅 랜딩페이지. 하늘 씨?

최하늘: 3월 런칭 캠페인용 랜딩페이지가 필요한데요. 카피는 제가 이번 주 금요일까지 준비할게요. 디자인은 이준호 님한테 부탁드려도 될까요?

이준호: 대시보드 디자인이 수요일에 끝나니까, 목요일부터 가능해요. 랜딩페이지는 이틀이면 될 것 같아요.

최하늘: 감사합니다! 그러면 다음주 금요일까지 랜딩페이지 완성 목표로 하겠습니다.

김서연: 좋습니다. 정리하면, 민수 씨 이번 주 결제 핫픽스, 준호 씨 수요일까지 대시보드 디자인, 하늘 씨 금요일까지 카피 준비. 다음 회의는 다음주 목요일 같은 시간에 할게요. 수고하셨습니다!
```

**Step 2-1**: 프로젝트 폴더에 회의 스크립트 파일 만들기

1. VSCode 왼쪽 파일 탐색기에서 **새 파일 아이콘(📄+)** 클릭
2. 파일 이름: `회의스크립트전문.md` 입력 후 Enter
3. 위에서 복사한 회의 스크립트 내용을 **붙여넣기** (Cmd+V)
4. **저장** (Cmd+S)

**Step 2-2**: 파일이 만들어졌는지 확인

VSCode 파일 탐색기에서 `회의스크립트전문.md` 파일이 보이면 OK!

---

**테스트 3**: 스킬로 회의록 요약해보기

이제 만든 스킬을 실제로 테스트합니다:

```
@회의스크립트전문.md 이 회의록 요약해줘
```

→ 스킬이 자동으로 실행되면 성공!

## 예상 결과물

이런 형태로 요약이 나오면 잘 된 겁니다:

```markdown
## 한 줄 요약
신규 대시보드 3월 첫째 주 출시 목표, 결제 핫픽스 우선 처리 후 진행

## 핵심 결정사항
- 차트 라이브러리: recharts 사용 (기존 스택 유지)
- 대시보드 3월 첫째 주 출시 목표
- CSV 내보내기: 대시보드 이후 3월 둘째 주 진행
- 마케팅 랜딩페이지: 다음주 금요일 완성 목표

## 액션아이템
| 담당자 | 할 일 | 마감 |
|--------|-------|------|
| 박민수 | 결제 모듈 핫픽스 | 이번 주 내 |
| 이준호 | 대시보드 UI 디자인 (recharts 기준) | 다음주 수요일 |
| 최하늘 | 랜딩페이지 카피 준비 | 이번 주 금요일 |
| 이준호 | 랜딩페이지 디자인 | 다음주 금요일 |
| 박민수 | 대시보드 개발 | 3월 첫째 주 |
| 박민수 | CSV 내보내기 개발 | 3월 둘째 주 |

## 다음 회의
- 다음주 목요일 14:00
```

## 💡 잘 만들어진 스킬의 특징

### description이 잘 써졌는가?

| 나쁜 예 | 좋은 예 |
|---------|---------|
| "회의록 요약" | "회의록을 요약합니다. '회의 요약해줘', '액션아이템 뽑아줘'라고 말하면 실행" |
| 한 단어 | **무엇을 하는지 + 언제 쓰는지** 모두 포함 |

> 🎯 **핵심**: description은 스킬의 **트리거 조건**입니다.
> 사용자가 말할 법한 키워드를 꼭 넣으세요!

---

### 본문이 명확한가?

| 나쁜 예 | 좋은 예 |
|---------|---------|
| "회의록을 잘 요약해주세요" | "1. 한 줄 요약 작성 2. 결정사항 추출 3. 액션아이템 표로 정리" |
| 애매한 지시 | **구체적인 단계**로 분해 |

> 🎯 **핵심**: Claude가 "이렇게 하면 되겠다"고 따라할 수 있게 쓰세요.

## ⚠️ 문제 해결

### "skill-creator가 안 돼요"

Claude에게:
```
skill-creator 스킬 설치해줘
```

### "스킬이 자동으로 안 불려요"

1. "사용 가능한 스킬 보여줘"로 설치 확인
2. description에 트리거 키워드 있는지 확인
3. 정확한 키워드 사용 (예: "회의 요약해줘")

### "만든 스킬 위치를 모르겠어요"

- 전역 스킬 위치: `~/.claude/skills/스킬이름/SKILL.md`
- Claude에게 "내 전역 스킬 폴더 위치 알려줘"라고 물어보세요
